# auto 分组重试策略

`auto` 是 API Key 的伪分组，请求进入 relay 前会按用户可用分组和令牌级 `auto_groups` 优先级解析为真实分组。

## 跨分组重试

令牌开启 `cross_group_retry` 后，如果当前真实分组的渠道返回可重试错误，系统会推进到下一个 `auto` 真实分组继续尝试。普通同分组重试仍受全局自动重试状态码规则约束：

- 允许：全局自动重试状态码内的错误，例如默认配置中的 `429`、`500`、`502`、`503`。
- 禁止：显式 `skip-retry` 错误、固定渠道请求、业务不可重试错误，以及 `504`、`524` 这类 always-skip 状态码。

`auto` 跨真实分组有一条窄口径网关例外：如果上游状态码是 `502`、`503` 或 Cloudflare `524`，系统允许推进到下一个真实分组。该例外只解决“第一个真实分组的上游网关坏了，`auto` 用户被锁死”的问题；它不改变普通同分组 retry 的 always-skip 规则，也不会绕过显式 `skip-retry`、固定渠道请求或渠道亲和的失败不切换保护。

`504` 仍保持 always-skip，不会因为 `auto` 跨分组重试而自动推进。若后续要把 `504` 也纳入网关例外，必须重新评估真实上游长任务、客户端超时和重复计费风险。

`RetryTimes` 仍控制普通同分组内的失败重试次数。为避免 `RetryTimes=0` 时 `auto` 请求被锁死在第一个真实分组，跨分组重试会在普通重试次数耗尽后单独判断是否还有下一个 `auto` 分组可用；有则重置本轮 retry 计数并切到下一真实分组。

## 渠道 setup 阶段失败

渠道选择完成后，系统还需要为选中的渠道准备上下文，例如读取可用 key、应用模型映射和渠道参数。对于可归因于渠道自身的 setup 错误，例如多 key 渠道已经没有任何启用 key，系统会把该渠道加入本次请求的排除集合并重新选路，而不是直接把 `channel:no_available_key` 返回给 `auto` 用户。

该重试不会绕过固定渠道请求，也不会绕过 relay 自引用循环保护。setup 阶段的渠道错误仍会上报到渠道可用性、熔断和自动禁用逻辑，避免后续请求继续优先选中同一个本地不可用渠道。

`auto` 选中真实分组后，会把本次请求的 `group` 上下文更新为真实分组，同时保留 `token_group=auto`。计费、日志和渠道亲和使用真实分组；跨分组重试仍根据 `token_group=auto` 判断该请求具备 auto failover 语义。

## 模型映射别名

渠道选择发生在 relay 阶段的 `model_mapping` 替换之前。因此 `auto` 能否选中渠道，取决于渠道能力表和内存缓存中是否存在请求里的模型名，而不是替换后的上游模型名。

渠道能力现在会同时包含：

- 渠道 `models` 中配置的模型名；
- 渠道 `model_mapping` 左侧的请求模型别名。

例如渠道 `models` 只包含上游真实模型 `anthropic.claude-sonnet-4-6`，同时配置 `model_mapping` 为 `{"claude-sonnet-4-6":"anthropic.claude-sonnet-4-6"}` 时，请求 `claude-sonnet-4-6` 可以先通过 `auto` 选中该渠道，然后在 relay 阶段映射为上游模型名。

无论 `MEMORY_CACHE_ENABLED` 是否开启，选路都会先尝试请求模型名，再尝试系统标准化后的模型名，例如 `gpt-4o-gizmo-*`、Gemini thinking 通配名等。关闭内存缓存的部署不应因为只配置了标准化模型能力而让 `auto` 分组误报没有可用渠道。

已有渠道在升级后如果仍然报“没有可用渠道”，需要重新保存相关渠道、按标签批量保存相关渠道，或在后台执行一次渠道能力修复，让 `abilities` 表和内存缓存按新规则重建。

## 初始令牌

新注册用户自动生成的初始令牌使用 `auto` 分组，并默认开启 `cross_group_retry`，与后台新建令牌弹窗的默认行为一致。

## 排查建议

如果 `auto` 分组请求持续返回 `status_code=502`、`status_code=503` 或 `status_code=524`：

1. 检查该 API Key 是否为 `auto` 分组。
2. 检查该 API Key 是否开启 `cross_group_retry`。
3. 检查 `系统设置 -> 分组与模型定价设置 -> 自动分组` 中至少有两个用户可用的真实分组。
4. 检查请求是否指定了固定渠道，或是否命中了要求失败不切换的渠道亲和缓存。
5. 检查全局“自动重试状态码”是否包含其它临时错误状态码。`502`、`503`、`524` 作为 auto 跨真实分组网关状态码时不依赖该列表即可推进，但普通同分组 retry 仍会按全局状态码和 always-skip 规则处理。
6. 如果错误是 `auto groups have no available channel for model ...`，检查展开出的真实分组内是否有启用渠道的 `models` 或 `model_mapping` 左侧别名覆盖该请求模型，并确认渠道能力已经重建。
7. 如果错误是 `channel:no_available_key`，检查该渠道是否为多 key 渠道且所有 key 都被自动禁用或手动禁用。正常情况下，只要同模型同分组还有其他可用渠道，本次请求会跳过该渠道继续选路；如果最终仍返回该错误，说明剩余候选渠道也不可用或已经被本请求排除。
8. 如果只剩一个真实分组或所有真实分组上游都返回 502/503/524，最终仍会把最后一次错误返回给用户。
