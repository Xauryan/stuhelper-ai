# auto 分组重试策略

`auto` 是 API Key 的伪分组，请求进入 relay 前会按用户可用分组和令牌级 `auto_groups` 优先级解析为真实分组。

## 跨分组重试

令牌开启 `cross_group_retry` 后，如果当前真实分组的渠道返回可重试错误，系统会推进到下一个 `auto` 真实分组继续尝试。可重试错误仍受全局自动重试状态码规则约束：

- 允许：全局自动重试状态码内的错误，例如默认配置中的 `429`、`500`、`502`、`503`。
- 禁止：显式 `skip-retry` 错误、固定渠道请求、业务不可重试错误，以及 `504`、`524` 这类 always-skip 状态码。

`RetryTimes` 仍控制普通同分组内的失败重试次数。为避免 `RetryTimes=0` 时 `auto` 请求被锁死在第一个真实分组，跨分组重试会在普通重试次数耗尽后单独判断是否还有下一个 `auto` 分组可用；有则重置本轮 retry 计数并切到下一真实分组。

## 初始令牌

新注册用户自动生成的初始令牌使用 `auto` 分组，并默认开启 `cross_group_retry`，与后台新建令牌弹窗的默认行为一致。

## 排查建议

如果 `auto` 分组请求持续返回 `status_code=502` 或 `status_code=503`：

1. 检查该 API Key 是否为 `auto` 分组。
2. 检查该 API Key 是否开启 `cross_group_retry`。
3. 检查 `系统设置 -> 分组与模型定价设置 -> 自动分组` 中至少有两个用户可用的真实分组。
4. 检查全局“自动重试状态码”是否包含对应临时错误状态码。
5. 如果只剩一个真实分组或所有真实分组上游都返回 502/503，最终仍会把最后一次错误返回给用户。
