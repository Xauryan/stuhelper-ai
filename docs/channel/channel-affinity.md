# 渠道亲和性策略

渠道亲和性会根据请求上下文、请求头或请求体中的 Key 复用上一次成功的渠道。该能力用于让同一会话、同一 trace 或同一业务标识尽量落到同一上游渠道，同时保留必要的故障切换能力。

## 临时错误突破亲和

后台入口：`系统设置 -> 运营设置 -> 渠道亲和性`。

新增全局配置：

- `fallback_on_temporary_error`：是否允许亲和渠道返回临时错误时突破亲和，默认 `true`。
- `temporary_error_status_codes`：被视为临时错误的 HTTP 状态码，默认 `429,500,502-503`。支持逗号分隔和范围，例如 `429,500,502-503`。

当某条亲和规则开启 `skip_retry_on_failure`，并且本次请求确实命中了亲和缓存渠道时，默认会阻止失败后切换渠道。开启 `fallback_on_temporary_error` 后，以下状态码会删除当前亲和缓存并允许走普通重试选路：

- `429`
- `500`
- `502`
- `503`

业务或认证类错误仍然保持原渠道亲和，不切换渠道，例如 `400`、`401`、`403`、`404`。

## 优先级

临时错误突破亲和不会绕过全局重试策略。满足以下任一条件时仍然不切换：

- 错误被显式标记为 `skip-retry`。
- 请求指定了固定渠道。
- 已无剩余重试次数。
- 状态码不在全局“自动重试状态码”范围内。
- 状态码属于 always-skip，例如 `504` 和 `524`。

因此，即使管理员把 `504` 或 `524` 写入 `temporary_error_status_codes`，系统仍会按 always-skip 规则处理，不会清除亲和并重试。
